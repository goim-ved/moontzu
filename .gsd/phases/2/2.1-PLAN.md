# Plan 2.1: Implement Subdomain Middleware

## Objective
Update the Next.js Middleware to handle dynamic subdomain routing. This allows the application to detect when it is being accessed via a tenant's subdomain (e.g., `tenant.localhost:3000` or `tenant.yoursaas.com`) and rewrite the request to a specific dynamic app route.

## Context
- .gsd/SPEC.md
- src/middleware.ts

## Tasks

<task type="auto">
  <name>Update Middleware for Subdomains</name>
  <files>src/middleware.ts</files>
  <action>
    - Modify `src/middleware.ts` to inspect the `request.headers.get("host")`.
    - Extract the subdomain (if any, ignoring `www` and the root domain).
    - If a valid subdomain exists, use `NextResponse.rewrite()` to route the request to `/app/[domain]${path}`.
    - If no subdomain exists, route normally to `/app/(main)${path}` or similar, or just let it pass if it's the root domain.
    - Ensure it still calls `updateSession(request)` for Supabase authentication.
  </action>
  <verify>Get-Content src/middleware.ts | Select-String "rewrite"</verify>
  <done>Middleware correctly identifies hostnames and rewrites routes while maintaining auth state.</done>
</task>

<task type="auto">
  <name>Setup Domain App Routes</name>
  <files>src/app/[domain]/page.tsx, src/app/[domain]/layout.tsx</files>
  <action>
    - Create a placeholder page at `src/app/[domain]/page.tsx`. This component should receive `params.domain` and display it indicating the middleware rewrite was successful.
    - Create a minimal layout `src/app/[domain]/layout.tsx` for tenant-specific pages.
  </action>
  <verify>Test-Path src/app/[domain]/page.tsx</verify>
  <done>Dynamic route folders for domains are created and accessible via middleware rewrite.</done>
</task>

## Success Criteria
- [ ] Middleware capable of hostname extraction.
- [ ] Next.js app routes incoming subdomain requests cleanly without visible URL redirects.
