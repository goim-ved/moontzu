# Plan 4.3: Stripe Checkout Integration

## Objective
Convert our public booking "Confirm Appointment" action into a flow that spins up a Stripe Checkout Session. The session routes the payment directly to the specific Tenant's connected Stripe account.

## Context
- src/app/[domain]/book/[serviceId]/page.tsx
- src/app/[domain]/book/actions.ts

## Tasks

<task type="auto">
  <name>Convert Action to Checkout Session</name>
  <files>src/app/[domain]/book/actions.ts</files>
  <action>
    - Update `createAppointment` logic. After confirming `tenant.stripe_account_id` exists:
    - Instead of immediately returning success, call `stripe.checkout.sessions.create()`.
    - Key parameters needed:
      - `line_items`: mapped from `service.price` and `service.name`.
      - `mode`: 'payment'.
      - `success_url` / `cancel_url`: Back to your domain.
      - `payment_intent_data.transfer_data.destination`: `tenant.stripe_account_id` (This routes the payout).
      - `metadata`: Store `customer_name`, `customer_email`, `start_time` so a webhook (future) or success callback can process the actual appointment creation if necessary. BUT for MVP without worrying about webhooks, we can do it synchronous: create the `appointment` as "pending", pass the `appointment_id` inside `client_reference_id` or `metadata` of the Session, and send the user to checkout.
    - Instead of `redirect('/success')`, redirect to `session.url`.
  </action>
  <verify>Get-Content "src/app/[domain]/book/actions.ts" | Select-String "stripe.checkout.sessions.create"</verify>
  <done>User is securely sent to Stripe Checkout with funds explicitly routed to the tenant's connected account.</done>
</task>

## Success Criteria
- [ ] Payments seamlessly flow from User -> System -> Connected Tenant using Destination Charges.
