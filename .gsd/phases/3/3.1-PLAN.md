# Plan 3.1: Database Schema Expansion

## Objective
Extend the Supabase multi-tenant schema to support the core booking engine features: defining services, staff availability schedules, and the actual customer appointments. Implement robust Row-Level Security (RLS) for absolute tenant isolation.

## Context
- .gsd/SPEC.md
- supabase/migrations/0000_core_setup.sql

## Tasks

<task type="auto">
  <name>Create Booking Schema Migration</name>
  <files>supabase/migrations/0001_booking_engine.sql</files>
  <action>
    - Create a new migration file `supabase/migrations/0001_booking_engine.sql`.
    - Create `services` table: `id`, `tenant_id` (refs tenants), `name`, `duration_minutes` (int), `price` (num), `created_at`.
    - Create `staff_availability` table: `id`, `tenant_id` (refs tenants), `user_id` (refs auth.users), `day_of_week` (int 0-6), `start_time` (time), `end_time` (time).
    - Create `appointments` table: `id`, `tenant_id` (refs tenants), `customer_name`, `customer_email`, `service_id` (refs services), `staff_id` (refs auth.users), `start_time` (timestamptz), `end_time` (timestamptz), `status` (text).
    - Enable RLS on all three tables `ALTER TABLE x ENABLE ROW LEVEL SECURITY;`.
  </action>
  <verify>Test-Path supabase/migrations/0001_booking_engine.sql</verify>
  <done>Booking tables defined properly with foreign key constraints mapped to tenants.</done>
</task>

<task type="auto">
  <name>Implement Booking RLS Policies</name>
  <files>supabase/migrations/0001_booking_engine.sql</files>
  <action>
    - Ensure tenant isolation utilizing the existing `get_user_tenants()` logic.
    - Owners/Admin/Staff can read/write their tenant's `services`, `staff_availability`, and `appointments`.
    - Public access: Anonymous/Users can ONLY READ `services` and `staff_availability` if they know the `tenant_id` (or leave it strictly based on server-side queries using service roles for public views). Since we query server-side via `supabase/server`, we don't necessarily need public unauthenticated RLS if we bypass with service_role, but standard is to allow anon SELECT where tenant matches `tenant_id`. Let's allow anon SELECT on `services` and `staff_availability` (for booking public pages), and anon INSERT on `appointments`.
  </action>
  <verify>Select-String -Path supabase/migrations/0001_booking_engine.sql -Pattern "CREATE POLICY"</verify>
  <done>RLS policies support secure internal dashboard views alongside public unauthenticated booking flow constraints.</done>
</task>
